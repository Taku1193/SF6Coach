# SF6成長支援ノートアプリ MVP仕様書

## 1. 概要

### 1.1 システム名
SF6成長支援ノートアプリ

### 1.2 目的
Street Fighter 6（以下、SF6）での上達を支援するため、ユーザーが対戦記録や動画要約をノートとして蓄積し、それらを参照しながらAIに相談できるWebアプリを提供する。

### 1.3 想定利用者
- SF6を継続的にプレイしている個人ユーザー
- 自分の使用キャラごとに学習内容を整理したいユーザー
- 対戦の反省や動画から得た知見を蓄積し、改善アクションにつなげたいユーザー

### 1.4 MVPの方針
- 現時点では主な利用者は1名でも、設計上は複数ユーザー利用を前提とする
- ユーザーごとのデータ分離を前提に、ノートデータには `userId` を保持する
- ベクトル検索や自動要約などの高度機能は含めず、ノートCRUDとAI相談を優先する

## 2. システム構成

### 2.1 全体構成
- フロントエンド: TypeScriptベースのWebアプリ
- 配信基盤: S3 + CloudFront
- バックエンド: AWS Lambda + API Gateway
- データベース: DynamoDB
- 認証基盤: Amazon Cognito User Pool
- AI連携: OpenAI API

### 2.2 構成上の制約
- OpenAI APIキーはフロントエンドに保持しない
- OpenAI APIの呼び出しはバックエンド経由で行う
- 静的サイトとして配信可能な構成とする
- 認証はメールアドレス + パスワード方式を採用し、フロントエンドは Cognito が発行したトークンを用いて API を呼び出す

## 3. 業務要件

### 3.1 基本ルール
- ノートは使用キャラごとに管理する
- ノート内容は、そのキャラ使用時の学習内容のみを扱う
- MVPでは1キャラ固定運用も許容するが、UI上はキャラ選択可能な構成が望ましい
- SF6は定期的に新キャラクターが追加されるため、キャラクター一覧をコードに強く埋め込まない構成とする
- 将来的なキャラクター追加時に、大規模な画面改修やデータ構造変更なしで対応できること
- キャラクターの表示名は、ユーザー混乱を避けるため可能な限りゲーム内呼称に合わせること

### 3.2 認証・ユーザー管理ルール
- ユーザーはメールアドレス + パスワードでサインアップおよびログインできること
- メールアドレス確認が完了したユーザーのみログイン可能とする
- パスワード再設定導線を提供すること
- ノート、AI相談、ユーザー設定相当のデータはユーザー単位で分離すること
- バックエンドで扱う `userId` は Cognito User Pool の `sub` を採用し、フロントエンド入力値で上書きしないこと
- API の認証可否判定はサーバー側で行い、未認証リクエストは処理しないこと

### 3.3 扱うノート種別
本システムで扱うノートは以下の2種類とする。

1. 対戦記録ノート
2. 動画要約ノート

## 4. 機能要件

### 4.0 認証機能
#### 概要
ユーザーがメールアドレス + パスワードで本人確認を行い、自身のノートデータへアクセスできるようにする。

#### 要件
- サインアップができること
- ログインができること
- ログアウトができること
- パスワード再設定ができること
- メールアドレス確認前のユーザーには確認完了を促すこと
- 認証済みセッションが有効な間は毎回の再ログインを強制しないこと
- 未認証時はノート画面、登録画面、AI相談画面、課題画面へ直接アクセスできないこと

### 4.1 キャラ選択機能
#### 概要
ログイン済みユーザーが使用キャラを選択し、そのキャラに紐づくノート管理画面へ遷移する。

#### 要件
- 使用キャラを1つ選択できること
- 選択したキャラを画面遷移やAPI呼び出し時の条件に利用できること
- MVPでは初期キャラ固定でもよいが、将来拡張しやすい構成とすること
- キャラクター選択肢は将来追加されるキャラクターを反映しやすい管理方式とすること
- 文字列として保持可能な設計とし、追加キャラクター投入時に既存ノート仕様を変更しないこと
- 画面上からキャラ選択画面へ戻れる導線をヘッダーに持てること

### 4.2 ノート一覧機能
#### 概要
選択中キャラに紐づくノートを一覧表示する。

#### 表示項目
- ノート種別
- タイトル
- タグ
- 更新日時
- お気に入り状態

#### 要件
- 一覧からノート詳細へ遷移できること
- 種別による絞り込みができること
- タグによる簡易絞り込みができること
- お気に入りのみ表示する絞り込みができること
- 各ノートの右下に星マークを表示し、ワンクリックでお気に入り登録・解除ができること
- お気に入り時の星マークは黄色の塗りつぶし、未お気に入り時は塗りつぶしなしで表示すること
- 更新日時の新しい順で表示することが望ましい

#### タイトル生成ルール
- 対戦記録ノート: `対戦相手キャラ + result` などを元に生成
  - 例: `JP戦 1-2 lose`
- 動画要約ノート: `videoTitle` をそのまま使用

### 4.3 対戦記録ノート機能
#### 概要
実際の対戦内容と反省点を記録する。

#### 登録・編集項目
- 使用キャラ
- 対戦相手キャラ
- 勝敗
- 良かったところ
- 改善点
- タグ

#### 機能
- 新規登録
- 一覧表示
- 詳細表示
- 編集
- 削除
- お気に入り登録・解除

#### バリデーション
- `opponentCharacter`: 必須
- `result`: 必須
- `goodPoints` と `improvements`: 両方空でも登録可能
  - ただしUI上はどちらか入力を促す

### 4.4 動画要約ノート機能
#### 概要
YouTube等の解説動画から得た知識を手動で要約して保存する。

#### 登録・編集項目
- 使用キャラ
- 動画タイトル
- URL
- 要約
- タグ

#### 機能
- 新規登録
- 一覧表示
- 詳細表示
- 編集
- 削除
- お気に入り登録・解除

#### バリデーション
- `videoTitle`: 必須
- `summary`: 必須
- `url`: 任意

#### 補足
- 動画取得、自動文字起こし、自動要約はMVP対象外

### 4.5 ノート詳細機能
#### 表示項目
- ノート種別
- 各項目の詳細
- タグ
- 作成日時
- 更新日時
- お気に入り状態

#### 機能
- 編集画面への遷移
- 削除実行
- 削除前確認ダイアログ表示
- 「編集」「削除」ボタンの隣に星マークを表示し、ワンクリックでお気に入り登録・解除できること
- お気に入り時の星マークは黄色の塗りつぶし、未お気に入り時は塗りつぶしなしで表示すること

### 4.6 AI相談機能
#### 概要
保存済みノートを参照し、ユーザーの相談内容に対して改善ポイントや次に意識すべき点を提案する。

#### 入力項目
- 相談内容（必須）
- 使用キャラ（必須）
- 相手キャラ（任意）
  - 共通キャラクター一覧からプルダウンで選択する
  - 未指定を選択できること
- タグ（任意）
- 参照ノート種別（任意）
  - `battleRecord`
  - `videoSummary`
  - 両方

#### 出力項目
- 相談内容の整理
- 改善ポイント
- 次に意識すること
- 参考にしたノート要素の要約またはタイトル一覧

#### バリデーション
- `consultationText`: 必須
- `character`: 必須

#### AIの役割
AIは以下を行う。
- 相談内容を整理する
- 関連ノートの内容を踏まえて改善点を提案する
- 次の練習・対戦で意識する内容を提案する

AIは以下を行わない前提とする。
- ノートに存在しない情報を断定する
- SF6固有の詳細フレーム知識を断定する
- 根拠のない攻略情報を強く提示する

#### 応答形式
バックエンドからの応答はJSON固定とする。

```json
{
  "summary": "相談内容の要約",
  "improvements": ["改善ポイント1", "改善ポイント2"],
  "nextActions": ["次に意識すること1", "次に意識すること2"],
  "referenceNotes": ["参考ノートタイトル1", "参考ノートタイトル2"]
}
```

### 4.7 課題管理機能
#### 概要
選択中キャラで直近の試合中に意識する課題を1件表示し、必要に応じて利用者が編集できるようにする。

#### 保存単位
- 課題は `ユーザー + 使用キャラ` ごとに1件のみ保持する
- 画面に表示する課題も常に1件のみとし、複数課題の履歴管理はMVP対象外とする

#### 表示項目
- 課題タイトル
  - 例: `対空を意識する`
  - 課題画面内で最も目立つ要素として大きく表示する
  - 改行を保持して表示できる
- メモ
  - 課題の理由、意識する場面、補足などを自由に記録できる
  - 課題タイトルの補足として大きめの文字で表示する
- 参考ノート
  - 対戦記録ノート、動画要約ノートの両方から選択できる
  - 最大3件まで選択できる
  - 選択済みノートは課題表示エリアの直下にリンクとして表示し、押下でノート詳細画面へ遷移できる
  - 参考ノート候補はタイトルで検索できる
  - 参考ノート候補は対戦記録、動画要約、すべてで種別フィルタできる

#### 編集項目
- 課題タイトル（必須）
- メモ（任意）
- 参考ノート（任意、最大3件）

#### 要件
- 選択中キャラに紐づく課題を取得して表示すること
- 課題未設定時は未設定であることを明示し、同じ画面から設定できること
- 課題タイトルは短い行動目標として視認しやすく表示すること
- 参考ノート選択時は、選択中キャラのノートから選べること
- 参考ノートには他ユーザーのノートを紐づけられないこと
- 削除されたノートが参考ノートに残っている場合は、表示時に欠落として扱うか非表示にできること

## 5. 関連ノート抽出仕様

### 5.1 基本方針
AIへ渡すノートは、バックエンド側のルールベースで抽出する。MVPではAIによる検索やベクトル検索は行わない。

### 5.2 抽出条件
優先度順に以下の条件を用いる。

1. 使用キャラ一致
2. 相手キャラ一致
3. タグ一致
4. ノート種別一致
5. 本文の簡易キーワード一致

### 5.3 抽出件数
- AIへ渡すノートは最大3〜5件程度
- 件数超過時は一致条件の多いものを優先する

### 5.4 スコアリング例
- 使用キャラ一致: 必須
- 相手キャラ一致: +3
- タグ一致: +2 × 一致数
- ノート種別一致: +1
- 本文キーワード一致: +1

## 6. 画面要件

### 6.1 画面一覧
1. ログイン画面
2. サインアップ画面
3. パスワード再設定画面
4. キャラ選択画面
5. ノート一覧画面
6. 対戦記録ノート登録/編集画面
7. 動画要約ノート登録/編集画面
8. ノート詳細画面
9. AI相談画面
10. 課題画面

### 6.2 UI/UX要件
- シンプルで視認性が高いこと
- スマホ/PCの両方で利用できること
- API通信中はローディング表示を出すこと
- エラー時はユーザーに理解しやすい文言を表示すること
- 削除時は確認ダイアログを表示すること
- 一覧、詳細、編集、登録の導線を明確にすること
- ヘッダーのアプリ名は、必要に応じて画面状態を初期化できる再読み込み導線として扱ってよい
- 認証切れ時はログイン画面へ戻し、再ログインを促すこと
- 課題画面では、試合中に確認しやすいように課題タイトルを最優先で大きく表示すること

## 7. データモデル

### 7.0 キャラクター管理方針
- `character` および `opponentCharacter` は文字列で保持する
- キャラクターIDの固定列挙に強く依存しない
- 将来的にはキャラクターマスタを別定義に切り出せる構成とする
- MVPではフロントエンド定数または設定ファイル管理でもよいが、追加時の変更箇所は最小限に抑える

### 7.1 共通型

```ts
type BaseNote = {
  noteId: string;
  userId: string;
  character: string;
  noteType: "battleRecord" | "videoSummary";
  isFavorite: boolean;
  tags: string[];
  createdAt: string;
  updatedAt: string;
};
```

### 7.2 対戦記録ノート

```ts
type BattleRecordNote = BaseNote & {
  noteType: "battleRecord";
  opponentCharacter: string;
  result: string;
  goodPoints: string;
  improvements: string;
};
```

### 7.3 動画要約ノート

```ts
type VideoSummaryNote = BaseNote & {
  noteType: "videoSummary";
  videoTitle: string;
  url: string;
  summary: string;
};
```

### 7.4 AI相談リクエスト

```ts
type AiConsultationRequest = {
  character: string;
  opponentCharacter?: string;
  consultationText: string;
  tags?: string[];
  noteTypes?: ("battleRecord" | "videoSummary")[];
};
```

### 7.5 AI相談レスポンス

```ts
type AiConsultationResponse = {
  summary: string;
  improvements: string[];
  nextActions: string[];
  referenceNotes: string[];
};
```

### 7.6 課題

```ts
type FocusIssue = {
  issueId: string;
  userId: string;
  character: string;
  title: string;
  memo: string;
  referenceNoteIds: string[];
  createdAt: string;
  updatedAt: string;
};
```

制約:
- `referenceNoteIds` は最大3件
- `title` は必須
- `memo` は空文字を許容する
- 同一 `userId + character` に対して有効な課題は1件のみ

## 8. API仕様

### 8.0 認証共通仕様
- ノート系 API、AI相談 API、課題 API は認証必須とする
- フロントエンドは `Authorization: Bearer <access token or id token>` ヘッダーを付与して API を呼び出す
- バックエンドでは認証トークンからユーザー情報を取得し、リクエスト本文の `userId` は信用しない
- 未認証または無効トークンの場合は `401 Unauthorized` を返す
- 認証済みでも他ユーザーの `noteId` へアクセスできないこと
- 認証処理には Cognito User Pool を利用する

### 8.1 ノート一覧取得
#### エンドポイント
`GET /notes?character=Luke&favoriteOnly=true`

#### 処理概要
- `character` に一致するノートを取得
- 認証トークンから取得した `userId` 条件を必ず付与
- `favoriteOnly=true` の場合はお気に入りノートのみ返却
- 一覧表示用データを返却

#### レスポンス例
```json
{
  "notes": []
}
```

### 8.2 対戦記録ノート登録
#### エンドポイント
`POST /notes/battle-record`

#### リクエスト例
```json
{
  "character": "Luke",
  "opponentCharacter": "JP",
  "result": "1-2 lose",
  "goodPoints": "対空は出ていた",
  "improvements": "ヴィーハト設置時に無策で突っ込みすぎた",
  "tags": ["JP", "対空", "設置対応"]
}
```

### 8.3 動画要約ノート登録
#### エンドポイント
`POST /notes/video-summary`

#### リクエスト例
```json
{
  "character": "Luke",
  "videoTitle": "JP初心者必見！",
  "url": "https://www.youtube.com/xxxx",
  "summary": "ルーキー〜ゴールド帯ではまずコンボ精度を上げる...",
  "tags": ["JP", "コンボ", "初心者"]
}
```

### 8.4 ノート更新
#### エンドポイント
`PUT /notes/{noteId}`

#### 処理概要
- 既存ノートを取得
- 種別に応じた更新可能項目のみ更新
- `updatedAt` を更新

### 8.5 ノートお気に入り更新
#### エンドポイント
`PATCH /notes/{noteId}/favorite`

#### 処理概要
- 該当ノートの `isFavorite` を更新
- 一覧画面、詳細画面の星マーク操作から利用する
- `updatedAt` を更新する

#### リクエスト例
```json
{
  "isFavorite": true
}
```

### 8.6 ノート削除
#### エンドポイント
`DELETE /notes/{noteId}`

#### 処理概要
- 該当ノートを削除
- 正常時は 200 または 204 を返却

### 8.7 AI相談
#### エンドポイント
`POST /ai-consultation`

#### リクエスト例
```json
{
  "character": "Luke",
  "opponentCharacter": "JP",
  "consultationText": "JP戦でヴィーハト設置後に毎回崩される。何を意識すべき？",
  "tags": ["設置対応"],
  "noteTypes": ["battleRecord", "videoSummary"]
}
```

#### レスポンス例
```json
{
  "summary": "JP戦で設置後の対応に課題があります。",
  "improvements": [
    "設置時に無理に前進しない"
  ],
  "nextActions": [
    "ヴィーハト設置時の様子見パターンを決める"
  ],
  "referenceNotes": [
    "JP戦 1-2 負け",
    "JP初心者必見！"
  ]
}
```

### 8.8 課題取得
#### エンドポイント
`GET /focus-issue?character=Luke`

#### 処理概要
- 認証トークンから取得した `userId` と `character` に一致する課題を取得する
- 参考ノートIDがある場合は、表示用にノートタイトル、ノート種別、更新日時を併せて返却する
- 課題未設定の場合は `issue: null` を返却する
- 参考ノート取得時も同一 `userId` のノートのみを対象とする

#### レスポンス例
```json
{
  "issue": {
    "issueId": "issue#Luke",
    "character": "Luke",
    "title": "対空を意識する",
    "memo": "前ジャンプを見たらまずしゃがみ強Pを出す。無理に前へ出ない。",
    "referenceNotes": [
      {
        "noteId": "note-1",
        "noteType": "battleRecord",
        "title": "JP戦 1-2 lose",
        "updatedAt": "2026-04-21T00:00:00.000Z"
      }
    ],
    "createdAt": "2026-04-21T00:00:00.000Z",
    "updatedAt": "2026-04-21T00:00:00.000Z"
  }
}
```

### 8.9 課題保存
#### エンドポイント
`PUT /focus-issue`

#### リクエスト例
```json
{
  "character": "Luke",
  "title": "対空を意識する",
  "memo": "前ジャンプを見たらまずしゃがみ強Pを出す。無理に前へ出ない。",
  "referenceNoteIds": ["note-1", "note-2"]
}
```

#### 処理概要
- 認証トークンから取得した `userId` と `character` に紐づく課題を upsert する
- `title` は必須、前後空白を除外した後に空なら `400 Bad Request` を返す
- `referenceNoteIds` は最大3件まで許可する
- 指定された参考ノートが同一ユーザーかつ同一キャラのノートであることを検証する
- 保存後は更新済みの課題を返却する

## 9. 非機能要件

### 9.1 性能
- 一覧取得や詳細取得は通常操作で待ち時間が大きくならないこと
- AI相談は数秒以内の応答を目標とするが、MVPではローディング表示で許容する

### 9.2 保守性
- フロントエンドとバックエンドの責務を分離する
- 型定義を明確にする
- 将来、embeddingや検索方式を差し替えやすい構成にする

### 9.3 セキュリティ
- OpenAI APIキーはサーバー側の環境変数で管理する
- 認証は Cognito User Pool を用い、JWT の検証結果に基づいて API 利用可否を判定する
- `userId` には Cognito User Pool の `sub` を保存し、フロントエンドから任意値を受け取らない
- 入力値バリデーションをバックエンドでも実施する

## 10. DynamoDB設計案

### 10.1 テーブル
`Notes`

### 10.2 主キー案
- PK: `userId`
- SK: `noteId`

### 10.3 属性
- `noteId`
- `userId`
- `character`
- `noteType`
- `tags`
- `createdAt`
- `updatedAt`
- `opponentCharacter` または `videoTitle` 等の種別固有属性
- 課題アイテムの場合は `issueId`、`title`、`memo`、`referenceNoteIds`

### 10.4 GSI案
- `GSI1PK`: `userId#character`
- `GSI1SK`: `updatedAt`

用途:
- キャラごとのノート一覧取得を効率化する

課題アイテムのキー方針:
- 課題は同一 `userId + character` で1件のみのため、`SK` は `focusIssue#${character}` のように固定値化する
- 既存ノート一覧に混ざらないよう、課題アイテムにはノート用の `noteType` を付与しない、または一覧取得時に除外できる種別を付与する
- 参考ノートは `referenceNoteIds` に既存ノートの `noteId` を保持し、表示時に必要なノート情報を取得する

必要に応じて将来追加:
- タグ検索用GSI
- 種別絞り込み用インデックス

## 10.5 キャラクター追加対応
- 新キャラクター追加時も `character` と `opponentCharacter` の保存形式は変更しない
- 既存データのマイグレーションなしで新キャラクター名を扱えること
- 追加対応は主にキャラクター選択肢の更新で完了する構成を目指す
- 将来的に外部設定ファイル、マスタAPI、DynamoDBマスタテーブル等へ切り出せるようにする
